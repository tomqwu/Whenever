## #7 — Second Cross-Check Provider (Kiwi/Tequila)

### Goal

Add a second real-fare provider so every matrix cell can be priced from an independent
source, and choose a concrete cross-check semantic.

### Design / Approach

**Recommended provider: Kiwi/Tequila.**
Skyscanner via RapidAPI is a viable alternative but RapidAPI rate limits and pricing tiers
add operational overhead for a small app. Kiwi/Tequila offers a free Self-Service tier with
a `v2/search` endpoint that returns round-trip itineraries, total prices, and deep booking links
— a natural fit for the existing normalized-dict contract.

**Cross-check semantic — recommended: ordered fallback (status quo), not true price comparison.**
The issue calls this "cross-checking" but there are two interpretations:

1. *Ordered fallback* (current behavior): `get_fare` tries each provider in order and returns
   the first real priced result. A second provider widens coverage — if Amadeus or Travelpayouts
   returns no data, Kiwi/Tequila fills the cell.

2. *True price comparison*: query all configured providers for the same cell, then surface the
   cheapest across providers (and optionally expose per-provider prices in the detail panel).

True comparison gives richer intelligence (the spec's `source` field already differentiates
providers) but roughly triples latency per cell for a matrix that may have 16–32 cells.
For the MVP the ordered-fallback approach is the right call: it closes coverage gaps without
multiplying latency. True comparison can be layered on later as an opt-in flag on `run_search`.

**Auth / env vars:**
```
KIWI_API_KEY=<token>           # from tequila.kiwi.com
```
`providers_configured()` in `app.py` should include `"kiwi"` when `KIWI_API_KEY` is set.
Add `KIWI_API_KEY` to `.env.example` and `docs/ARCHITECTURE.md`'s config table.

**Normalized fare dict shape (unchanged):**
```python
{
    "cheapest_cad": int,   # total for party (CAD)
    "stops":        int,   # max stops across outbound + return legs
    "nonstop_cad":  int | None,
    "source":       "kiwi",
    "book":         str | None,  # deep link from Kiwi response
}
```

**Cache key interaction:** No change needed. The cache key in `_fare_cache` is
`(origin, dest, dep, ret, adults, children)` — provider-agnostic. Since `get_fare`
returns the first real result from whichever provider answers, the cached value already
carries `source` so the caller always knows where the price came from.

### Files to Create / Modify

- `app.py` — add `kiwi_fare(origin, dest, dep, ret, adults, children) -> dict | None`;
  insert it into the provider tuple inside `_get_fare_uncached`; extend `providers_configured()`.
- `.env.example` — add `KIWI_API_KEY=`.
- `docs/ARCHITECTURE.md` — add `KIWI_API_KEY` row to the config table; update fare-adapter prose.
- `CONTRIBUTING.md` — no structural change; the "Adding a flight provider" steps already
  describe the pattern exactly.
- `tests/test_app.py` (or `tests/test_providers.py`) — new unit tests for `kiwi_fare`.
- `tests/e2e/` — extend the provider-health e2e to assert `"kiwi"` appears when the env var is set.

### Task Breakdown

1. Explore Kiwi/Tequila `v2/search` docs; confirm round-trip, CAD currency, and deep-link
   field names. Draft the normalized mapping.
2. Implement `kiwi_fare` with error handling (non-200, empty data, missing fields).
3. Extend `providers_configured()` and add `KIWI_API_KEY` to `.env.example`.
4. Write unit tests: `kiwi_fare` happy path (mock `requests.get`), no-data path,
   HTTP error path; `providers_configured` includes `"kiwi"` only when key is set.
5. Update `docs/ARCHITECTURE.md` config table and fare-adapter diagram.
6. Run `pytest --cov=app --cov=cli --cov-fail-under=99` and resolve any coverage gaps.

### Test Strategy

- Mock `requests.get` at the unit level — never call the live Kiwi API in tests.
- Fixture a realistic Kiwi JSON response (a handful of itineraries with `price`, `transfers`,
  `deep_link` fields) and assert `kiwi_fare` normalizes it correctly.
- Test the `no-key` path: with `KIWI_API_KEY` unset, `kiwi_fare` returns `None`.
- Test the fallback chain: patch `amadeus_fare` and `travelpayouts_fare` to return `None`;
  assert `get_fare` falls through to `kiwi_fare` and returns its result.
- E2E: run `GET /api/health` with `KIWI_API_KEY` set in the test environment and assert
  `"kiwi"` appears in the `providers` list.
- Coverage gate: new provider function must be ≥99% covered; the cache interaction is already
  covered by existing `_fare_cache` tests — no additional work needed there.

### Risks / Constraints

- **Real-data-only guardrail:** `kiwi_fare` must never return a fabricated price. If Kiwi
  returns an unexpected shape, log a warning and return `None` rather than guessing.
- **Amadeus decommission risk:** The spec notes Amadeus Self-Service is decommissioned on
  July 17, 2026. After that date `amadeus_fare` will fail silently and Kiwi/Tequila becomes
  the primary provider; having it in place before the cutoff is important.
- **CAD currency support:** Confirm Kiwi/Tequila supports `currency=CAD` on their endpoint.
  If only USD is available, apply an exchange rate or document the gap.
- **Rate limits:** Kiwi free tier has per-minute and per-day limits. The existing `_fare_cache`
  (TTL 3600s) already absorbs repeated lookups for the same cell; document the limit in
  `docs/ARCHITECTURE.md`.

## #8 — "Watch This Trip" Daily Price-Drop Alerts

### Goal

Let users persist a saved search and receive a notification when the best fare for that
trip drops below their last-seen price.

### Design / Approach

**Persistence: SQLite (via stdlib `sqlite3`).**
A JSON flat-file is simpler to start but fragile for concurrent writes and harder to query
when multiple watches exist. SQLite with a two-table schema (`watches`, `price_history`) is
a single-file store, no extra dependency, and straightforward to test. Schema:

```sql
CREATE TABLE watches (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    origin      TEXT NOT NULL,
    dest_iata   TEXT NOT NULL,
    dest_city   TEXT,
    dep_date    TEXT NOT NULL,   -- ISO date
    ret_date    TEXT NOT NULL,
    adults      INTEGER NOT NULL DEFAULT 2,
    children    INTEGER NOT NULL DEFAULT 0,
    threshold_pct REAL NOT NULL DEFAULT 25.0,
    last_price  INTEGER,         -- last known cheapest_cad
    last_source TEXT,
    created_at  TEXT NOT NULL,
    active      INTEGER NOT NULL DEFAULT 1
);

CREATE TABLE price_history (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    watch_id    INTEGER NOT NULL REFERENCES watches(id),
    checked_at  TEXT NOT NULL,
    price       INTEGER,
    source      TEXT,
    book        TEXT
);
```

The DB file path is configured via `WATCH_DB` env var (default `whenever_watches.db`
in the working directory).

**Scheduler: CLI subcommand run by cron.**
A Flask background thread or APScheduler adds dependency weight that is not justified
for a small app. Instead, `cli.py` gains a `whenever watch` sub-interface:

```
whenever watch add   --from YYZ --city PEK --dep 2026-12-14 --ret 2027-01-04 [--adults 2] [--child 11]
whenever watch list
whenever watch check        # re-prices all active watches; emits drops to stdout + optional webhook
whenever watch remove <id>
```

The user (or a cron job) runs `whenever watch check` on a schedule. This is deliberately
simple and avoids a long-running daemon.

**Notification mechanism — recommended MVP: stdout log + optional webhook POST.**

| Mechanism   | MVP | Later |
|-------------|-----|-------|
| Stdout/log  | yes | yes   |
| Webhook POST | yes (WATCH_WEBHOOK_URL env var) | yes |
| Email (SMTP) | no | yes   |
| Push / SMS  | no | later |

On a price drop, `watch check` prints a structured line:
```
[PRICE DROP] PEK  YYZ→PEK  2026-12-14/2027-01-04  CA$7,890 → CA$6,950  (-$940)  book: https://...
```
If `WATCH_WEBHOOK_URL` is set, it also POSTs a JSON body with the same fields. Email is
deferred — too much SMTP configuration overhead for MVP.

**Price-drop definition:** a new `cheapest_cad` that is strictly less than `last_price`
stored in the `watches` table. No minimum delta is enforced in the MVP; add a `--min-drop`
flag later if users need noise filtering.

### Files to Create / Modify

- `watch.py` (new) — `WatchDB` class wrapping `sqlite3`; `check_all_watches(db)` function
  that calls `app.get_fare` per watch and compares against `last_price`; webhook POST helper.
- `cli.py` — add `watch` subcommand group using argparse subparsers; wire `add`, `list`,
  `check`, `remove` to `watch.py` functions.
- `app.py` — no changes required; `watch.py` imports and calls `get_fare` directly.
- `pyproject.toml` — add `"watch"` to the `py_modules` list (currently `["app", "cli"]`) so
  the installed package includes `watch.py`. If a `whenever-watch` console-script entry-point
  is desired (e.g. `whenever-watch = "watch:main"`), add it to `[project.scripts]` here.
- `.env.example` — add `WATCH_DB=` and `WATCH_WEBHOOK_URL=`.
- `docs/ARCHITECTURE.md` — add a "Price Watch" section describing the DB file, cron pattern,
  and notification options.
- `tests/test_watch.py` (new) — unit tests for `WatchDB` and `check_all_watches`.
- `tests/e2e/test_watch_cli.py` (new) — e2e tests invoking `whenever watch` subcommands.

### Task Breakdown

1. Design and create `watch.py`: `WatchDB` (init, add, list, remove, update_price), and
   `check_all_watches` (iterates watches, calls `app.get_fare`, detects drops, returns report).
2. Add the `watch` subcommand group to `cli.py` (subparsers: `add`, `list`, `check`, `remove`).
3. Implement webhook POST in `watch.py`; add `WATCH_WEBHOOK_URL` to `.env.example`.
4. **Update `pyproject.toml`:** add `"watch"` to `py_modules`; optionally add a
   `whenever-watch` console-script entry-point pointing at `watch:main`.
5. Write unit tests for `WatchDB` using an in-memory SQLite DB (`:memory:`);
   test `check_all_watches` with a mocked `get_fare`.
6. Write e2e tests: `whenever watch add ...` → `whenever watch list` → `whenever watch check`
   (with `get_fare` monkeypatched to return a lower price) → assert drop is reported.
7. Update `docs/ARCHITECTURE.md`; update `CONTRIBUTING.md` with the `WATCH_DB` env var.
8. Coverage pass: `pytest --cov=app --cov=cli --cov=watch --cov-fail-under=99`.

### Test Strategy

- All SQLite operations tested with `":memory:"` so tests leave no files on disk.
- `check_all_watches` is tested by injecting a watch with `last_price=8000`, monkeypatching
  `app.get_fare` to return `cheapest_cad=7000`, and asserting the function returns a drop record.
- Test the no-drop path (same price) and the no-data path (`cheapest_cad=None`).
- Webhook POST: mock `requests.post` and assert it is called with the expected body on a drop;
  assert it is NOT called when there is no drop.
- Coverage note: `watch.py` must be added to the `pytest --cov` module list.

### Risks / Constraints

- **Real-data-only guardrail:** `check_all_watches` calls `app.get_fare`, which already
  enforces the real-provider chain. No mock prices must reach the stored `last_price`.
- **Cache interaction:** `get_fare` with `FARE_CACHE_TTL=3600` will serve a cached price
  for up to an hour. Cron should run `whenever watch check` no more than once per hour,
  or the operator should set a shorter TTL for the watch runner process. Document this.
- **Concurrent writes:** `sqlite3` in WAL mode handles light concurrent access safely.
  For a single-user or low-frequency cron setup this is fine. Document the limitation.
- **Cross-cutting effect on `run_search`:** none — `watch.py` bypasses `run_search` and
  calls `get_fare` directly (the watch is a single dep×ret cell, not a matrix).

## #15 — PDF / CSV Export of the Fare Matrix

### Goal

Export the full `run_search` output (summary cards + per-city dep×ret matrix) to a PDF
report and a CSV file, matching the reference `china-flight-matrix.pdf` described in the spec.

### Design / Approach

**Libraries:**

| Format | Recommended library | Why |
|--------|--------------------|----|
| PDF    | `weasyprint`       | HTML-to-PDF; GUI and PDF share the same HTML/CSS template, matching the spec's note that the reference PDF was generated from Chromium |
| CSV    | stdlib `csv`       | No extra dependency; sufficient for flat matrix rows |

`reportlab` (direct PDF drawing API) is more powerful but forces maintaining a separate
layout system from the web UI. `weasyprint` is the right call: render a dedicated Jinja2
template to HTML and pass it through WeasyPrint. The template can reuse the matrix CSS
already in `templates/index.html` but is a standalone file for print layout control.

**PDF content (per the spec's reference report):**
- Title: origin, destination, date window, passengers, cabin, currency, freshness timestamp.
- Search settings and legend (stop labels, highlight definitions).
- Summary winner cards: cheapest, recommended, low-stop winner, simple upgrade.
- One dep×ret matrix section per arrival city (same structure as `run_search` `grid`).
- Cell content: price (CAD), stop count, chosen flag (nonstop `*` or cheapest).
- Footer: provider coverage note, stale-price warning, real-data disclaimer.

**CSV content:** one row per `(city, dep_date, ret_date)` cell, columns:
`city, iata, dep_date, ret_date, cheapest_cad, stops, nonstop_cad, chosen, chosen_cad, source, book`.

**Where it hooks in:**

- *Web endpoint:* two stateless POST routes — the client re-posts the same search parameters
  it sent to `/api/search`, the server runs `run_search` and streams the file response directly.
  There is no server-side job storage and no `GET /api/export/<job_id>` route.
  (If caching export results by job ID is wanted in the future, add a "persist search jobs"
  prerequisite task before introducing any job-ID scheme.)
  - `POST /api/export/pdf` — accepts the same JSON body as `POST /api/search`; runs
    `run_search`, renders `templates/export_matrix.html`, passes to
    `weasyprint.HTML.write_pdf()`, returns `application/pdf`.
  - `POST /api/export/csv` — accepts the same JSON body as `POST /api/search`; runs
    `run_search`, writes CSV rows, returns `text/csv; charset=utf-8`.

- *CLI flag:* add `--export-pdf <path>` and `--export-csv <path>` to `cli.py`'s main
  parser. After `run_search` returns, call the export helpers from a new `export.py` module.

**Module boundary:** extract export logic into `export.py` (new) so it is independently
testable and shared by the web and CLI surfaces.

### Files to Create / Modify

- `export.py` (new) — `render_pdf(result: dict) -> bytes` (HTML → WeasyPrint bytes);
  `render_csv(result: dict) -> str` (CSV text).
- `templates/export_matrix.html` (new) — standalone print-layout Jinja2 template for the PDF.
- `app.py` — add `POST /api/export/pdf` and `POST /api/export/csv` routes; call `export.py`.
  No `GET /api/export/<job_id>` routes are added (stateless POST-only design; see above).
- `cli.py` — add `--export-pdf` and `--export-csv` flags; call `export.py` after `run_search`.
- `pyproject.toml` — add `"export"` to the `py_modules` list (currently `["app", "cli"]`) so
  the installed package includes `export.py`.
- `requirements.txt` — add `weasyprint`.
- `tests/test_export.py` (new) — unit tests for `render_pdf` and `render_csv`.
- `tests/e2e/test_export.py` (new) — e2e: POST to `/api/export/pdf` and assert `Content-Type`.

### Task Breakdown

1. Create `export.py` with `render_csv(result) -> str`; test with a fixed `run_search`-shaped
   dict; confirm all matrix columns are present.
2. Create `templates/export_matrix.html` with print-optimized layout (title, legend, per-city
   matrix tables).
3. Implement `render_pdf(result) -> bytes` using WeasyPrint; add `weasyprint` to
   `requirements.txt`.
4. Wire `POST /api/export/pdf` and `POST /api/export/csv` into `app.py` (stateless POST-only;
   no job-ID routes).
5. Add `--export-pdf` and `--export-csv` flags to `cli.py`; write to the given file paths.
6. **Update `pyproject.toml`:** add `"export"` to `py_modules` so the installed package
   includes `export.py`.
7. Write unit tests for `render_csv` (header row, data rows, no-data cells) and `render_pdf`
   (assert bytes starts with `%PDF`).
8. Write e2e tests for both export endpoints (monkeypatch `run_search` with a fixture dict
   to avoid live API calls in CI).

### Test Strategy

- `render_csv`: supply a hand-crafted `run_search` output dict; assert the CSV has the right
  header row and that price/source/book fields map correctly; test a `chosen_cad=None` cell
  renders as empty string, not a crash.
- `render_pdf`: assert output is bytes and starts with `b"%PDF"` (WeasyPrint produces valid PDF).
  Full visual correctness is a manual check; the automated gate is structure-level.
- Web endpoints: monkeypatch `app.run_search` to return a fixture dict; assert status 200 and
  correct `Content-Type`; assert `Content-Disposition: attachment` header for the download.
- CLI flags: call `main(["--from","YYZ","--city","PEK",...]`)` with `--export-csv /tmp/out.csv`;
  assert the file exists and parse the CSV; same for `--export-pdf`.
- **Real-data-only guardrail:** export routes call `run_search` which calls `get_fare`. In
  unit tests, monkeypatch `get_fare` to return fixture dicts. In e2e tests, monkeypatch
  `run_search` directly. Never supply mock prices as user-visible output outside tests.

### Risks / Constraints

- **WeasyPrint system dependencies:** WeasyPrint requires `libpango`, `libcairo`, and related
  system libs. Document OS prerequisites in `CONTRIBUTING.md`; add a CI step to install them
  (e.g., `apt-get install -y libpango-1.0-0 ...` in GitHub Actions).
- **Latency:** `POST /api/export/pdf` re-runs the full matrix search before exporting. For
  large grids (6 cities × 4×4 dates = 96 cells) this is slow. The MVP accepts this; a future
  job-ID approach would let the export reuse a cached result.
- **Cross-cutting effect on `run_search`:** none — `export.py` consumes the dict that
  `run_search` already returns; no changes to `run_search` are needed.

## #16 — Configurable Country-Seed List

### Goal

Replace the pure-LLM country expansion in `top_cities` with a config-driven seed list
(LLM as fallback only), and expose user-facing include/exclude controls.

### Design / Approach

**Config format: a YAML file (`config/country_seeds.yaml`).**
JSON is also viable but YAML is more readable for a city/airport list that humans will
hand-maintain. The schema:

```yaml
china:
  display_name: "China"
  candidates:
    - city: "Beijing"
      iata: "PEK"
      alt_iata: ["PKX"]
      priority: 1
      notes: "Primary gateway; strong direct-route supply from YYZ/YVR"
    - city: "Shanghai"
      iata: "PVG"
      alt_iata: ["SHA"]
      priority: 2
    - city: "Guangzhou"
      iata: "CAN"
      priority: 3
    - city: "Shenzhen"
      iata: "SZX"
      priority: 4
    - city: "Chengdu"
      iata: "TFU"
      alt_iata: ["CTU"]
      priority: 5
      notes: "CTU coverage varies by provider"
    - city: "Xiamen"
      iata: "XMN"
      priority: 6
    - city: "Haikou"
      iata: "HAK"
      priority: 7
      optional: true
    - city: "Sanya"
      iata: "SYX"
      priority: 7
      optional: true
    - city: "Shenyang"
      iata: "SHE"
      priority: 8
      optional: true
      notes: "Family/region-specific"
```

Country keys are normalized lowercase (matches `top_cities(country, n)` after `country.lower()`).
The `optional: true` flag marks cities that are excluded from the default `n` unless the user
opts in.

**Integration with `top_cities`:**

Current: `top_cities(country, n)` — pure LLM call, cached by `lru_cache`.

New behavior:
1. Normalize `country` to lowercase; look up in the loaded seed config.
2. If a seed entry exists: return ALL candidates (both required and optional) sorted by
   `priority`, each annotated with `optional: bool`. The caller decides which to surface;
   `top_cities` itself does not filter by `optional`.
3. If no seed entry exists: fall back to the existing LLM path (`ollama_chat`); returned
   entries have `optional: False` (LLM-sourced cities are all considered required).

The `lru_cache` on `top_cities` remains valid — the seed lookup is deterministic.
There is no `include_optional` parameter on `top_cities`: the function always returns the
full candidate set and lets the call site (web route or CLI) decide what to show.

**User include/exclude (web + CLI):**

- Web: `/api/top-cities` returns ALL candidates (required + optional), each with `optional`
  and `priority` fields. Example response entry:
  ```json
  {"city": "Haikou", "iata": "HAK", "priority": 7, "optional": true}
  ```
  The frontend renders required cities pre-selected and optional cities as an opt-in
  checklist. `POST /api/search` already accepts `destinations` as a list of `{city, iata}`
  — the user's final selection (after checking/unchecking) is the filter mechanism; no
  additional API change is needed.
- CLI: add `--include-optional` flag to `cli.py` (`--country` mode only). When absent, the
  CLI silently drops entries where `optional=True` from the resolved destination list before
  passing them to `run_search`. When present, all candidates are included.
  A future `--exclude <IATA>` flag can filter the list further.

**Config loading:** load `config/country_seeds.yaml` at module import time in `app.py`
using `PyYAML` (`yaml.safe_load`). Cache the parsed result in a module-level dict
`_SEED_CONFIG`. If the file is absent, log a warning and fall back to LLM-only mode.

### Files to Create / Modify

- `config/country_seeds.yaml` (new) — seed data starting with China (per the spec's list).
- `app.py` — load `_SEED_CONFIG` at startup; update `top_cities` to check seeds first and
  return ALL candidates annotated with `optional: bool` (no `include_optional` parameter —
  the function always returns the full set). Update the `/api/top-cities` route to pass
  through `optional` and `priority` fields in its JSON response so the frontend can render
  the opt-in checklist without any additional query parameters.
- `cli.py` — add `--include-optional` flag; when absent, filter out `optional=True` entries
  from the `top_cities` result before passing to `run_search`.
- `requirements.txt` — add `pyyaml` (likely already present; confirm).
- `docs/ARCHITECTURE.md` — document the seed config path and fallback behavior.
- `CONTRIBUTING.md` — document how to add a country seed.
- `tests/test_app.py` — unit tests for `top_cities` with seed present and absent;
  test `include_optional` behavior.
- `tests/e2e/test_top_cities.py` — e2e: `POST /api/top-cities` with `country=China` returns
  Beijing, Shanghai, etc. from seeds without an LLM call.

### Task Breakdown

1. Create `config/country_seeds.yaml` with the China seed (all 9 candidates from the spec).
2. Add `_SEED_CONFIG` loading to `app.py`; handle missing file gracefully.
3. Refactor `top_cities` to check `_SEED_CONFIG` first; return ALL candidates (required +
   optional) annotated with `optional: bool`; no `include_optional` parameter on the function.
   The existing `lru_cache` key `(country, n)` remains valid — the return value always
   contains the full set.
4. Update `/api/top-cities` route to pass through `optional` and `priority` per city in the
   JSON response. Required cities (`optional: false`) should be pre-selected in the UI; optional
   cities (`optional: true`) should appear as an opt-in checklist.
5. Add `--include-optional` flag to `cli.py`; when absent, filter `optional=True` entries
   from the resolved list before passing to `run_search`.
6. Write unit tests: seed present → LLM not called; seed absent → LLM called;
   `top_cities` always returns optional cities in the result set; CLI without
   `--include-optional` drops them; CLI with `--include-optional` includes them.
7. Update `docs/ARCHITECTURE.md` and `CONTRIBUTING.md`.
8. Coverage pass: `pytest --cov=app --cov=cli --cov-fail-under=99`.

### Test Strategy

- Monkeypatch `_SEED_CONFIG` directly in tests — no file I/O needed in unit tests.
- Test `top_cities("China", 6)`: assert the LLM (`ollama_chat`) is NOT called when a seed
  exists; assert the returned list matches the top 6 priority-ordered candidates.
- Test `top_cities("Narnia", 6)`: assert the LLM IS called (seed absent).
- Test that `top_cities` always returns optional cities in the result set (the function does
  not filter by `optional`); assert each entry has `optional: bool` set correctly.
- Test CLI filtering: with `--include-optional` absent, assert optional cities are excluded
  from the list passed to `run_search`; with `--include-optional` present, assert they are
  included.
- E2E: `POST /api/top-cities {country: "China"}` with `OLLAMA_HOST` pointing at a non-existent
  server — should still return all seed cities including optional ones (not error out), proving
  the LLM is not invoked. Assert each optional city has `"optional": true` in the JSON response.
- `lru_cache` invalidation: call `top_cities.cache_clear()` in `setUp`/fixture to avoid
  cross-test contamination.

### Risks / Constraints

- **LLM fallback guardrail:** The LLM never produces prices. The seed-vs-LLM split is only
  for city/IATA resolution, which is already LLM territory. No conflict with the real-data
  guardrail.
- **`lru_cache` stability:** `top_cities` retains its `(country, n)` signature — no
  `include_optional` parameter is added to the function. The cache key is unchanged.
  Optional-city filtering happens at call sites (CLI flag, future UI controls), not inside
  `top_cities`, keeping the cache valid and avoiding any hashability concerns.
- **Cross-cutting effect on `run_search`:** `run_search` receives `dests` as an already-resolved
  list; it does not call `top_cities`. The change is fully contained in `top_cities` and the
  two call sites (`api_top_cities` route and `cli.py`).
- **Config drift:** As providers evolve, `alt_iata` fields may need updating (e.g., Chengdu
  TFU vs CTU). Document the config file as the authoritative source and note that IATA codes
  must be verified against the configured provider.

## Sequencing

These four features touch the codebase at different layers. Most are independently shippable:

| Feature | Primary files touched | Can start immediately? |
|---------|-----------------------|------------------------|
| #7 Kiwi provider | `app.py` (`_get_fare_uncached`, `providers_configured`) | Yes — isolated function addition |
| #8 Price watch | `watch.py` (new), `cli.py` (new subcommand) | Yes — calls `get_fare` but does not modify it |
| #15 PDF/CSV export | `export.py` (new), `app.py` (new routes), `cli.py` (new flags) | Yes — consumes `run_search` output without modifying it |
| #16 Country seeds | `app.py` (`top_cities`), `config/` (new) | Yes — contained in `top_cities`; `run_search` is unaffected |

**Recommended order:**

1. **#7 first.** The Amadeus Self-Service decommission on July 17, 2026 is a hard deadline.
   Getting Kiwi/Tequila in place before that date ensures the app does not lose its only
   real-fare provider. This is time-sensitive; do it before any of the others.

2. **#16 next.** Country seeds are small (one YAML file + a `top_cities` refactor) and
   directly ground the MVP in the spec's candidate list. It also unblocks the spec's
   "arrival-city candidate checklist" use case referenced by the export (#15) and watch (#8)
   features (which benefit from having well-known city/IATA pairs).

3. **#8 and #15 in parallel** once #7 and #16 are merged. Price watch and export do not
   share any files beyond the `run_search` output dict. Two developers (or two branches)
   can work on them simultaneously.

No feature serializes on another except the practical priority of #7 ahead of the
Amadeus deadline.
